# 我怎麼處理 AI 長任務的「斷層」

一篇與領域無關的短文，說明這套格式為什麼存在。

---

## 問題通常不在模型

用 AI 做多天任務時，摩擦很少是「寫不出來」，更常是：

- 換一個 session，不知道上一段做到哪
- 對話越長，越容易偏題、重複、悄悄改方向
- 重要決定散在聊天裡，過幾天很難找回
- 版本管理記得住檔案 diff，但記不住**任務脈絡**

我把這叫做 **長任務斷層**——發生在模型、IDE、代理之間的縫隙，而不是某一個工具不夠強。

---

## 聊天紀錄不適合當任務資料庫

聊天很適合**探索**，不適合當**唯一真相來源**：

| 需要記住的 | 聊天紀錄的問題 |
|-----------|-------------|
| 做到哪一步 | 難搜、難比對 |
| 為什麼這樣決定 | 埋在長對話裡 |
| 下一步可能做什麼 | 容易被 AI 自行延伸 |
| 哪些還沒驗證 | 容易遺失 |
| 怎麼交給下一個 session | 要重講整段故事 |

解法不是「再寫一個更好的 prompt」，而是長任務需要一種**可交接的狀態格式**。

---

## 我的做法：任務狀態膠囊

在明確里程碑，我把任務狀態從對話裡抽出來，壓成一小組固定檔案——**Task State Capsule（任務狀態膠囊）**。

產品不是新 App，產品是**固定的形狀**：

```text
進度報告      → 進度、阻塞、下一步
決策紀錄      → 選擇、理由、曾考慮的替代方案
分支資訊      → 主線 / 實驗 / 回滾 / 審查 的命名規則
接手指令      → 下一個 AI session 可直接貼上的 prompt
復原檢查表    → AI 跑偏時要核對什麼
狀態清單      → 版本 hash、時間戳、機器可讀 metadata
```

概念上，這是 session 之間的 **Interlock Layer（互鎖交接層）**：

```text
Session A 做到里程碑 → 更新膠囊 → Session B 從膠囊接手
```

對話是**工作過程**，膠囊是**換班快照**。

---

## 通用工作流程

1. 在明確節點存檔——不是每句話都存
2. 花幾分鐘更新膠囊，而不是在新對話裡重講半小時脈絡
3. 下一個 session 貼接手指令，並附上膠囊
4. 若範圍開始膨脹，跑復原檢查表，而不是從頭爭論

格式是**工具中立**的：模型、IDE、代理可以換，膠囊形狀不必跟著換。

---

## 三層分工

| 層 | 記什麼 |
|----|--------|
| 版本管理（如 Git） | 程式碼與檔案變更 |
| 任務膠囊 | 進度、決策、下一步、邊界 |
| 審計封包（可選） | 這版交付能不能被驗證 |

膠囊不取代 Git，也不匯出 raw chat。它保存的是**任務還在進行中的狀態**。

---

## 一個直覺

AI 越強，**交接與驗證**越重要——不是越不重要。

生成變快之後，成本常常轉移到：

- 怎麼延續
- 怎麼限界
- 怎麼證明「這步算完成」

所以我不再問「下一個 session 還記不記得」，  
改問：**「我手上有沒有值得交接的任務狀態？」**
